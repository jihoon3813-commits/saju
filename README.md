# 🔮 Dream & Fortune (꿈과 운의 사전) — 서비스 운영 및 환경 설정 매뉴얼

본 문서는 전통 동양 명리학 알고리즘과 최신 생성형 AI(Gemini 2.5) 레이어 및 78장 타로 물리 덱 엔진이 상호 유기적으로 결합된 프리미엄 운세 서비스의 개발, 배포, 장애 대응 및 운영 기준서입니다.

---

## 1. 환경 구성 및 로컬 실행 (Local Execution)

### 1) 선수 환경 요구사항
- **Node.js**: v20.x 이상 권장
- **NPM**: v10.x 이상
- **Database**: 로컬 개발 시 내장 JSON DB 파일 (`local_db.json`)을 자동으로 인메모리 파싱하여 동작하므로 PostgreSQL 없이도 즉각 구동 가능합니다. (실제 운영 환경은 PostgreSQL을 환경변수로 연계하여 구동합니다.)

### 2) 의존성 설치 및 로컬 서버 기동
```bash
# 의존 패키지 다운로드
npm install

# 로컬 개발용 turbopack 서버 구동 (기본 포트: 3000)
npm run dev
```
- 브라우저로 `http://localhost:3000`에 접속하여 개인화 홈과 운세 서비스를 이용할 수 있습니다.

## 2. 배포 및 데이터베이스 관리 (Deployment & DB Migration)

### 1) 빌드 및 배포
프로덕션 컴파일 최적화 및 린트/타입 엄격 검사를 수행하여 static 에셋을 생성합니다:
```bash
# 빌드 수행
npm run build

# 배포 패키지 테스트 구동
npm run start
```

### 2) DB 마이그레이션 (Database Migration)
- **migrations 폴더 위치**: [`/migrations`](file:///d:/anti-gv/30. 운세/migrations)
- **규칙**: 신규 기능 개발에 따른 테이블 확장 시 순차적으로 번호가 매겨진 DDL 파일(예: `0007_phase8_extensions.sql`)을 작성합니다.
- **실행 원리**: 서비스 구동 시 초기화 파일인 `init.ts`가 가동되어 미적용 SQL 마이그레이션 스크립트를 식별하여 데이터베이스 풀에 자동으로 실행(Auto DDL migration)하므로 별도의 수동 커맨드 실행 없이 안전하게 동기화됩니다.

### 3) 데이터 시드 (Database Seeding)
- **시더 파일 위치**: [`src/lib/db/seeder.ts`](file:///d:/anti-gv/30. 운세/src/lib/db/seeder.ts)
- **시딩 정보**: 
  - 기본 가입 사용자 및 비로그인 관리자 계정 모의
  - P0 무료 4대 상품 및 P1 확장 유료 상품 14종 정보 자동 세팅
  - 3대 법적 정책 문서 (이용약관, 개인정보처리방침, 쿠키 방침) 1.0.0 버전 적재
  - 사이트 내 8대 광고 슬롯 배치 기본 명세 시딩
- **시드 실행**: 빌드 중 또는 서버 첫 부팅 시 데이터베이스에 정보가 전무한 경우 `init.ts`에 의해 백그라운드로 1회 자동 호출됩니다.

## 3. 환경변수 상세 명세 (Environment Variables)

배포 및 실행 환경에 따라 아래 변수들을 `.env.local` 또는 클라우드 플랫폼(Vercel, AWS) 대시보드에 기입하여 관리합니다.

| 환경변수명 | 설명 | 기본값 / 권장값 | 필수 여부 |
|---|---|---|---|
| `NODE_ENV` | 실행 모드 분기 (개발, 테스트, 프로덕션) | `development` / `production` | 필수 |
| `DATABASE_URL` | PostgreSQL DB 연결 주소 (운영 환경용) | `postgresql://user:pass@host:5432/db` | 선택 (미지정시 JSON DB 구동) |
| `GEMINI_API_KEY` | Gemini AI 분석용 구글 개발자 API 키 | `AIzaSy...` (가짜 키 입력 금지) | 필수 |
| `NEXT_PUBLIC_ENABLE_ADS` | 전역 광고 슬롯 활성화 토글 (Feature Flag) | `false` (안전 배포) / `true` (광고 ON) | 필수 |
| `JWT_SECRET` | 사용자 인증 세션 토큰 암호화 키 | `any-random-secret-string` | 필수 |
| `NEXT_PUBLIC_BASE_URL` | 캐노니컬 SEO 및 메타링크용 베이스 URL | `http://localhost:3000` | 필수 |

## 4. 명리학적 계산 기준 (Calculation Basis)

본 서비스의 만세력 엔진은 객관적 절입 계산과 경도 차이를 엄밀히 보정하여 천간/지지를 추출합니다.

- **표준시 기준 (KST)**: 한국 표준시의 기점인 동경 135도 자오선을 기저로 삼아 양/음력 캘린더 변환을 수행합니다.
- **진태양시 보정 (True Solar Time)**: 
  - 출생한 실제 지역의 경도(Longitude)에 입각하여 태양 남중 고도 시차 분을 연산하여 최종 시주(Hour Pillar)를 보정합니다.
  - 예: 서울(동경 126.97도) 출생자는 도쿄(동경 135도 기준시)보다 약 32분 늦은 태양 시간을 가집니다. (옵션 활성화 시 자동 반영)
- **야자시(夜子時) 및 조자시(朝子時) 경계 조율**:
  - 일간의 변화 기준은 23:30분을 경계선으로 나뉩니다.
  - 23:00 ~ 23:30분에 태어난 경우 조자시 규정을 적용받으며, 23:30 ~ 00:00분에 출생한 경우 야자시로 분류되어 **일주(Day)는 당일을 유지하고 시주(Hour)만 자(子)시**로 산출합니다.
- **시간 미상 처리**: 출생 시간을 모르는 상태(`unknownBirthTime: true`)로 입력 시 시주(Hour Pillar)는 `null` 처리되며 명리 원국의 천간/지지 오행 분포는 년, 월, 일 3개 기둥(6글자)만으로 계산됩니다.

## 5. AI 프롬프트 및 룰셋 버전 운영 (AI Prompts & Rules)

AI 해석의 일관성과 윤리 조항 확보를 위해 프롬프트 컴파일러와 룰셋 버전을 엄격히 격리 운영합니다.

- **서비스 레지스트리 ([serviceRegistry.ts](file:///d:/anti-gv/30. 운세/src/lib/ai/serviceRegistry.ts))**: 사주 상품 18종의 분석 섹션과 입력 필수값 목록을 명세화하여, 프롬프트 빌더가 분기 지옥 없이 해석 양식을 동적 컴파일하도록 제어합니다.
- **윤리 규칙 강제 조항**: 시스템 프롬프트 컴파일러 내부에는 **"사망 연도, 치료 불가능한 질병, 극단적 사고, 100% 수익 보장 재테크"**에 대한 단정적 예측을 원천 불허하는 윤리 규칙이 결합되어 있어 오남용을 상시 통제합니다.
- **버전 표기법**:
  - `Engine Version`: v1.0.0 (만세력 공식 계산 모듈)
  - `Rule Version`: v1.0.0 (십신별 룰기반 설명 구조)
  - `Prompt Version`: v1.0.0 (Gemini 2.5 프롬프트 양식)

## 6. 장애 대응 및 폴백 가이드 (Emergency Response)

AI API 제한이나 네트워크 장애 발생 시 사용자 이탈을 예방하기 위해 룰 기반의 요약 엔진(Rule-Based Fallback)이 활성화됩니다.

- **장애 감지**: Gemini API 호출 실패, 파싱할 수 없는 비정형 문항 출력, 또는 안전 안심 필터링에 의한 수신 거부(3회 재시도 실패) 시 즉시 폴백 모드가 발동됩니다.
- **폴백 로직**: 만세력 계산식에서 출력된 오행 분포 수치, 십신 대조표, 지지 결합 및 신살 데이터를 토대로 이미 내부 정의된 데이터베이스 룰 해설 문서가 실시간 조합되어 사용자에게 최종 반환됩니다.
- **대응 매뉴얼**: 
  - AI 응답 지연 누적 시, 어드민에서 룰셋 버전을 점검하거나 `.env`의 API 키 유효성을 확인하십시오.
  - 어드민 로그에 찍힌 감사 로그 및 사용자 신고 큐(`userReports`)를 통해 이상 징후를 모니터링할 수 있습니다.

## 7. 백업·복구 및 개인정보 파기 (Backup, Recovery & Privacy)

### 1) 백업 및 복구 절차
- **JSON 파일 백업**: 로컬 파일 `local_db.json`을 일 단위 cron 스케줄로 복제 보관합니다.
- **Postgres 백업**:
  ```bash
  # 백업 실행
  pg_dump -U username -h host -d dbname -F c -b -v -f backup_file.dump
  
  # 복구 실행
  pg_restore -U username -h host -d dbname -v backup_file.dump
  ```

### 2) 개인정보 파기 및 마스킹 처리법
- **수동 삭제 요청**: 사용자가 보관한 사주 프로필의 즉각적인 영구 삭제 요구 시, 관리자는 관리 메뉴에서 해당 프로필 ID의 `deletedAt` 필드를 현재 시간으로 업데이트(Soft Delete)하고, 요청 시 DB 원장에서 아예 물리적으로 `DELETE` 처리합니다.
- **신고 마스킹**: 사용자가 입력 폼 오류 및 환불 신고(`/api/user/reports`) 시 기입한 본문 정보는 DB 적재 전 필터에 의해 신용카드 번호, 이메일, 전화번호가 `[CARD_MASKED]`, `[PHONE_MASKED]`로 변경 마스킹되어 보관 기간(30일) 경과 시 자동 파기 대상이 됩니다.

## 8. 결제·환불 및 광고 통제 가이드 (Payment & Ads)

### 1) 결제 및 환불 워크플로
- **결제 공급자**: 포트원 및 토스페이먼츠 연동 규격을 제공합니다. 개발계 배포 시에는 Mock PG 인터페이스를 통해 가상 테스트 결제창이 나타납니다.
- **수동 환불**: 주문 관리 대장에서 환불 요청 건을 식별한 후, 관리자 권한을 통해 `triggerAdminRefundAction`을 구동하면 PG사 환불 API 연계와 동시에 주문의 상태가 `refunded`로 강제 변경되고 결과 접근 권한이 즉시 정지됩니다.

### 2) 광고 차단 배포 가이드 (Ad-Blocker friendly)
- 실제 광고 태그나 스크립트가 사이트 레이아웃의 CLS(누적 레이아웃 이동)를 일으키는 것을 막기 위해, 시딩된 높이값(`reserveHeight`)만큼 광고 슬롯 공간이 상시 확보됩니다.
- 아직 광고 승인이 나지 않은 상태에서는 **`NEXT_PUBLIC_ENABLE_ADS=false`**로 환경변수를 설정하고 배포하십시오. 광고 슬롯 자체가 마운트되지 않아 사용자 액션(결제, 생년월일 기입)을 전혀 방해하지 않는 클린 상태로 우선 출시할 수 있습니다.

## 9. 콘텐츠 발행 워크플로 (Content Publishing)

1. **원고 작성**: 에디터/리뷰어 권한을 소유한 필진이 `/admin/content` 지면에서 마크다운 형태로 운세 백과 정보나 칼럼 작성.
2. **리뷰 및 승인**: 상태값을 `draft`에서 `review`를 거쳐 `published`로 승인 변경.
3. **색인 자동 갱신**: 신규 칼럼이 최종 발행되면 `/sitemap/contents.xml` 정적 색인 파일에 신규 URL이 자동 추가되어 포털(네이버, 구글) 로봇이 즉시 색인할 수 있도록 반영됩니다.
